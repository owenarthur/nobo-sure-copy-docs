Security — Sure Copy

Effective date: 4 July 2026

Vendor: nobo.dev  ·  Security contact: support@nobo.dev

Architecture: Runs on Atlassian

Sure Copy is a Forge app that runs entirely on Atlassian's infrastructure. All of its compute and storage are Atlassian-hosted. It qualifies for Atlassian's Runs on Atlassian program, which means:

  • Zero data egress. Your content never leaves Atlassian. The app makes no calls to any external server, third-party service, analytics provider, or remote API.
  • No remote infrastructure. There are no servers, databases, or logs operated by us outside of Atlassian's platform.
  • No credentials collected. The app never asks for, receives, or stores Atlassian passwords, API tokens, or personal access tokens (PATs). It authenticates through Atlassian's Forge platform using granted OAuth scopes.

What data the app handles

  • Your Confluence content, in transit only. During a copy, the app reads pages, attachments, and labels from the source space and writes them to the destination — all through Atlassian's Confluence APIs, entirely within your Atlassian site. It does not retain copies of your content.
  • Minimal job metadata. To run and resume copy jobs, the app stores small records in Atlassian's Forge storage (KVS): the Atlassian account ID of the user who started the job, source/destination identifiers, page IDs and titles, and job status. This never leaves Atlassian and can be deleted from the app's job history.
  • Anonymous usage counts. The app keeps simple aggregate counters (number of jobs, pages, attachments copied) in Forge storage. These are counts only — they contain no page content and no personal data.

Access & permissions (least privilege)

The app requests only the scopes it needs to copy content, and no administrative scopes:

  • read/write:space:confluence — read the source space and create the destination space.
  • read/write:page:confluence — read source pages and create the copies. Blog posts share this same scope (Confluence treats them as page-type content), so the optional "Copy blog posts" feature needs no extra scope: it reads the source blog posts and recreates them, with their bodies, attachments and labels, in the new space.
  • read/write:attachment:confluence and write:confluence-file — copy page attachments.
  • read/write:label:confluence — copy page labels.
  • read/write:folder:confluence — copy folders (the organisational containers that group pages in the tree) and re-create the source's folder nesting in the destination, so copied pages land under the right folders. Read-side reads folder titles and their place in the tree; write-side re-creates them. No page content flows through this scope.
  • read/write:template:confluence — copy the source space's user-created page templates (name, body, description, labels) to the new space, so a team cloning a space to reuse its templates gets them. Read-side lists the source's templates with their bodies; write-side recreates them in the destination. It applies only when copying into a brand-new space and only when the source has templates; no page content flows through these scopes. (The space theme/look-and-feel is not copied — Confluence Cloud has no usable API for it; see the limitations.)
  • read:content-details:confluence — read-only. Count the content types the app doesn't copy yet (whiteboards, databases) so the completion report can tell you exactly what was left behind. No content is read or moved by this scope; it is used only to inventory and report.
  • read/write:comment:confluence — for the optional "Copy footer comments" feature only. When you turn that option on for a copy, the app reads the source pages' footer (page) comments and recreates them on the copies. The option is off by default, so no comment is read or written unless you opt in per copy. Inline comments are never copied.
  • write:content.restriction:confluence — the one optional power scope, for the optional "Recreate page restrictions" feature only. It is the single scope that touches access control, so it stays behind a toggle that is off by default: with it off, no restriction API call is made at all. When you turn it on for a copy, the app reads each source page's view/edit restrictions (reading needs no extra scope) and applies the equivalent to the copy — the same users and groups. The Sure Copy app account is added to each restricted copy so it doesn't lock itself out; this grants no additional human any access. Any restriction principal it can't reproduce is reported on the job, never silently dropped. When the toggle is on, the pre-copy check additionally confirms that you hold the right to set restrictions on the source, so the app never uses this scope to escalate a view-only user into one who can lock down pages.
  • storage:app — store job state in Atlassian's Forge storage.

Before any copy runs, the app verifies — as the requesting user — that they are permitted to read the source and create the destination (and, when "Recreate page restrictions" is on, that they may set restrictions on the source). It does not escalate a user's privileges.

Data retention & deletion

Job records live in Atlassian's Forge storage for your site. You can delete an individual job record from the app's history at any time. When the app is uninstalled, its Forge storage is removed by Atlassian per the Forge platform's data-retention behavior.

Platform security & maintenance

The app relies on Atlassian's Forge platform for its runtime, storage, network isolation, and platform-level patching. Application dependencies are kept current and reviewed for known vulnerabilities. All communication with Confluence happens over Atlassian's platform (TLS-protected).

Reporting a security issue

Please report suspected security issues to support@nobo.dev. We aim to acknowledge reports within a few business days.
