Troubleshooting

The five things most likely to bring you to this page, with what's actually happening and what to do. If yours isn't here, email support@nobo.dev with the source and destination space keys and when you started the copy.

"My copy looks stuck / is taking a long time"

First: is it actually stuck? The progress view shows three useful numbers — the page counter ("Creating pages: 47 of 151"), the elapsed time, and the engine invocations counter. The copy runs as a background job in small time-budgeted batches, so the invocations counter should tick up steadily. A large space is simply a lot of batches: expect a few hundred pages to take minutes, not seconds, and thousands of pages to take a while — with the progress bar moving the entire time. Remember the bar covers *two* passes: all pages are created first, then every page is rewritten, so 50% means "creation done, rewriting starting".

Rate limiting looks like a stall and isn't. If Confluence rate-limits the app, the job saves its state and reschedules itself after the delay Confluence requests — that can be up to 15 minutes of silence, after which it resumes on its own. During this window the counters don't move. This is the app behaving correctly inside Confluence Cloud's rules, not a failure.

When to actually worry: if the page counter *and* the invocations counter haven't moved for well over 15 minutes, something's genuinely wrong. If the job ends in failed, you'll get the error text and a Retry button — retrying is safe (see the last section on this page). If it sits in a running phase without progressing and never fails, contact support with the space keys; there's currently no user-facing cancel or force-resume for a job that's running, and we'd rather look at it than have you guess.

"A link in my copy points at the old space"

Work through these in order — the first three are the app working as designed:

1. Is the link's target inside the copied set? Links to pages in *other* spaces, or to anything not part of this copy, are preserved untouched on purpose — they still point where they should. Only links between copied pages get rewritten.

2. Is it a /x/ short link? Short links are opaque codes and are never rewritten. Each one was recorded as a warning on the job (e.g. 'Team charter': shortlink: not rewritten (resolve upstream): /wiki/x/AbCdE), so you have a precise list of where they are. Replace them with normal page links in the copy if they matter.

3. Is the URL inside a code block? URLs inside code macros and preformatted content are treated as content, not navigation, and left alone by design.

4. Does the source space have two pages with the same title? Title-based links to a duplicated title are rewritten with an ambiguous: duplicate title warning recorded — check the linked page is the one you expect.

5. None of the above? Then we want to know. A link between two copied pages that still points at the source, with no warning recorded, is a rewriting miss — we treat those as our highest-severity class of bug. Email support@nobo.dev with both page titles and the link URL; include the space keys so we can find the job record and its warnings.

"Something didn't get copied"

Two different mechanisms, two different answers:

Per-page issues (shown in the progress view). If an individual page, attachment, or label operation fails mid-copy, the job records it and carries on — one poisoned page never kills the job. The issues panel lists entries like:

  • create 'Page title': … — the page itself couldn't be created; it and its content are missing from the copy.
  • rewrite 'Page title': … — the page exists in the copy but its body may still contain unrewritten links (the rewrite pass failed on it).
  • attachment somefile.pdf: … — that one file didn't make it; the page did.
  • labels: … — a page's labels weren't applied.

The panel shows the first 25 issues; up to 50 are stored per job (beyond that, a counter records how many more were dropped). For most issues, fixing the odd page by hand in the copy is faster than re-running; for a systemic failure (dozens of identical errors), contact support with the error text.

Unsupported content types (not shown — check the source). Whiteboards, databases, folders, blog posts, inline and footer comments are not copied in the current version, and — honestly — the current report does not itemise them. The full list is in Limitations; if your source space leans on those types, audit against that list rather than the report.

"I get a permission error before the copy starts"

The exact message: You don't have permission to view space "KEY" (or it doesn't exist). Ask a space admin for access, then try again.

Before creating a job, the app checks — as you, with your permissions — that you can view the source space. This is deliberate: the app never starts a space copy for someone who can't see that space. If you hit this, either the space key doesn't exist or your account lacks view permission on it; a space admin can grant access under Space settings → Permissions.

Two neighbouring messages worth knowing:

  • A space with key "KEY" already exists. Choose a different key. — every copy targets a brand-new space; pick an unused key. This is also what you'll see if you try to start the same copy twice: the first job took the key.
  • Space key must start with a letter and contain only letters and numbers. / Destination key must differ from the source space key. — key validation, exactly as stated.

"The copy was interrupted — will retrying duplicate everything?"

No. This is the scenario the engine was specifically built for. When a job fails you'll see the failure panel with the error and this promise, which we can back up: *"Retrying resumes safely: pages that were already copied are reused, not duplicated."*

How it actually works: the job's map of *source page → copy* survives the failure, and every page-creation step checks whether the page already exists in the destination before creating anything. Pressing Retry rewinds the job to the start; it re-walks the space, finds each already-copied page, reuses it, creates only what's missing, and then re-runs the link-rewrite pass (which is idempotent — rewriting an already-rewritten page changes nothing). The pages/attachments/labels counters reset and recount during the retry so the final numbers are accurate for the finished copy.

Fine print, in the interest of honesty:

  • Retry appears only on failed jobs. A job that's still in a running phase can't be retried (Job is "create", not "failed" — nothing to retry.) — because it's still running.
  • Attachments are re-processed on retry. Where the copy already has a file with the same name, Confluence may reject the duplicate upload; any such rejection shows up as a per-attachment issue. The already-uploaded file is unaffected — nothing is lost, the issue entry is just noise from the overlap.
  • The per-page issues list is cleared on retry and repopulated by the new run, so what you see at the end describes the final state, not the failed attempt.

To verify after a retry: the completion page count should equal the source space's page count, and spot-check that no page appears twice in the new space's page tree. It won't.

When you contact support

Include: the source and destination space keys, roughly when the job started, and — if it failed — the error text from the failure panel. That's enough for us to find the job record, which contains the per-page issues and every rewrite warning.

If we need the app's execution logs for your site: Forge app logs are gated by *you*, not us — a Confluence site admin can grant the developer temporary log access from Admin → Connected apps (Atlassian calls this app log access). We'll only ask when it's genuinely needed, and you can revoke it afterwards.
