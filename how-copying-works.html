How copying works

This page explains what Sure Copy actually does when it copies content — in particular how link rewriting works, because that's the difference between a copy that works and a copy that quietly points back at the original. No prior knowledge of Confluence internals required.

Sure Copy runs the same engine in three modes: copying a whole space into a brand-new space; copying a page tree into an existing space (under a parent page you choose); and copying a single page (with or without its descendants), which you can also launch from the "Copy page tree to…" action on any page's ••• menu. The phases and the link rewriting below are identical across all three — the only difference is which pages are in the copied set and where they land. (The examples on this page describe a whole-space copy; substitute "the copied set" for "the space" for the other modes.)

The short version

A copy runs in three phases:

  1. Preparing — the app reads the full page list of the source space and creates the destination space.
  2. Creating pages — every page is recreated in the new space in hierarchy order (parents before children), along with its attachments and labels. The app keeps a running map of source page → its copy.
  3. Rewriting links — each copied page's content is then written in full, with every internal reference re-pointed at the copies using that map.

The job runs in the background on Atlassian's infrastructure, in short batches, and records exactly what it did — including anything it chose not to rewrite, and why.

Every release of Sure Copy has to pass an automated end-to-end correctness battery: real copies of a fixture space on a live Confluence Cloud site, verified check by check — attachments, macros with references, legacy URLs, awkward titles, deep nesting, large bodies. If a check fails, the release doesn't ship.

Why links break in naive copies

Here's the problem every copy tool has to solve, and most don't.

Confluence stores page content in an XML-based "storage format". A link to another Confluence page isn't stored as visible text — it's stored as a reference, in one of several shapes:

  • A page reference: "the page titled Budget 2026 in space MKT". This is what you get when you type /link or paste a page link — and it's also how macros like Include Page and Excerpt Include point at their source pages.
  • A URL: /wiki/spaces/MKT/pages/12345/Budget-2026, where 12345 is the page's unique ID.
  • A legacy URL: /wiki/display/MKT/Budget+2026, the older title-based format that still lives in thousands of long-lived spaces.
  • A short link: /wiki/x/AbCdE, an opaque code that Confluence resolves server-side.

Notice what all of these have in common: they name the original space or the original page ID. A naive copy — one that just duplicates page content byte for byte — preserves every one of these references verbatim. The copy looks perfect. Then someone clicks a link, and they're silently back in the source space. Every internal link, every Include Page macro, every image that references an attachment on another page: all still wired to the original.

This is the single most commonly reported failure of space and page-tree copies in Confluence, and it's insidious because nothing looks broken. Readers just quietly end up in the wrong space — or, if the source space is later archived or deleted, at a dead link.

What Sure Copy rewrites

During the copy, the app builds an ID map: every source page ID paired with the ID of its copy, plus the set of page titles included in the copy. In the rewrite phase, it parses each page's storage format with a real XML parser — not find-and-replace — and walks every reference:

  • Page references (ri:page elements) whose space is the source space and whose title is in the copied set have their space key flipped to the new space. This works wherever the reference appears — in a link, inside a macro body such as Include Page or Excerpt Include, or inside a cross-page attachment reference — regardless of attribute order or entity-encoded titles.
  • Page URLs (/spaces/SRC/pages/12345/...) are rewritten to the new space key and the new page ID, using the ID map. This applies to URLs in link attributes and to URLs pasted as bare text in the page body. The slug and any #anchor suffix are preserved untouched.
  • Legacy /display/SRC/Page+Title URLs are rewritten to the new space key when the title (after decoding the +-and-percent encoding) is in the copied set. Anchors survive here too.
  • Cross-page attachment references — an image on page A that displays a file attached to page B — are fixed by the same page-reference rewrite, so they resolve to page B's copy.
  • /x/ short links (/wiki/x/AbCdE) are resolved to the page they target: each in-set page carries its own short-link code (returned by the Confluence API alongside the page), so the app maps the code back to the source page and, if that page is in the copied set, rewrites the short link to the canonical copy URL. A short link to a page outside the copied set is left as-is and warned (see below). Resolutions are cached per copy.

The rewriting is surgical: only the exact span being changed is touched. Everything else in the page — attribute order, whitespace, entity encoding, macro structure — survives byte for byte. And it's idempotent: running the rewrite over its own output changes nothing, which matters for resume.

What Sure Copy deliberately leaves alone

Just as important as what gets rewritten is what doesn't. These are judgment calls, and here's each one:

  • Links to pages outside the copied set. If a page links to another space, or to a page that isn't part of this copy, the link still points where it should — so it's left exactly as-is. Rewriting it would break it.
  • External URLs. Absolute URLs to other sites (or anything that isn't clearly a link within your Confluence site) are never touched.
  • Code blocks and preformatted content. A URL someone typed inside a code macro is content, not navigation. It's left alone by design.
  • /x/ short links to pages outside the copied set. In-set short links are now resolved and rewritten to the copy (see What Sure Copy rewrites above). A short link whose target is not part of this copy — another space, or a page not included — is left exactly as-is, with a warning recorded on the job so you know where it is. Rewriting it would break it.
  • Duplicate titles. Confluence Cloud can allow two pages with the same title, which makes title-based references ambiguous. Sure Copy rewrites the reference and records an "ambiguous: duplicate title" warning against the page so you can spot-check it.

The warnings system

Anything the rewriter skipped or was unsure about is recorded as a warning on the job, tagged with the page title it occurred on — for example:

  • 'Team charter': shortlink: target not in copied set — left as-is: /wiki/x/AbCdE
  • 'Roadmap': skipped: ri:page title not in copied set — left pointing at MKT: "Archive index"
  • 'Budget 2026': ambiguous: duplicate title in copied set — ri:page link rewritten by title: "Notes"

The philosophy: when in doubt, don't touch it — and say so. A link that's cautiously left pointing at the source is recoverable; a link that's confidently rewritten to the wrong place is not. Warnings are stored with the job record (up to 100 per job), and per-page errors are shown directly in the progress view. If you ever find an internal link that should have been rewritten and wasn't — with no warning — we treat that as our highest-severity bug. Email support@nobo.dev.

Homepage handling: merge vs nest

Confluence automatically creates a homepage for every new space, so a space copy has to decide what to do with the source space's homepage. Sure Copy gives you the choice:

  • Merge (default). The source homepage's content is written into the new space's automatically created homepage, links rewritten like any other page. The destination homepage keeps its own title (which matches the new space's name). Every link that pointed at the source homepage now points at the destination homepage. Result: one homepage, no stray duplicate. This is right for almost everyone.
  • Nest. The source homepage is copied as its own page, placed as a child of the new space's homepage, keeping its original title. Pick this if the source homepage is a real content page in its own right and you want it preserved verbatim as a page.

Large spaces: batching, progress, and resume

Confluence Cloud sets firm limits on how long any single background operation may run, and rate-limits API traffic. Sure Copy is built for that reality rather than around it:

  • Time-budgeted batches. The job processes pages until it approaches a ~15-second budget, saves its state, and re-queues itself. A large space is hundreds of these small batches, each one checkpointed. The "engine invocations" counter in the progress view is this mechanism made visible.
  • Rate limiting is handled, not fatal. If Confluence returns a rate-limit response, the job saves its state and reschedules itself after the delay Confluence asks for (up to a 15-minute pause). This looks like the copy going quiet for a while; it is not a failure. It resumes on its own.
  • Every create is checked first. Before creating a page, the app checks whether that page already exists in the destination. Combined with the persistent ID map, this makes the whole engine resume-safe: if a job is interrupted and retried, already-copied pages are found and reused — never duplicated. (The exact retry semantics are in Troubleshooting.)
  • One bad page never kills the job. If an individual page fails to copy — an exotic macro, a permission anomaly — the error is recorded against that page and the job carries on. The progress view lists every page-level issue.

Progress is polled every few seconds; the progress bar spans both page walks (creating, then rewriting), so 50% means "all pages created, rewriting about to begin".

What to check after your first copy

  1. Open the new space's homepage — the source homepage content should be there (under merge, the default).
  2. Open two or three pages that link to each other. Hover the links: the URLs should contain the new space key and new page IDs.
  3. Open a page with images and attachments — they should render from the copy, not the source.
  4. Check the completion summary against expectations: pages, attachments, labels counts, and the page-issues panel if present.

If anything in that list surprises you, Troubleshooting covers the likely causes — and support@nobo.dev covers the rest.
