Limitations

What Sure Copy does not copy (yet) — and how you'll know.

We list these up front because silent data loss is worse than a documented limitation. Some of the things below are deliberate design decisions; some are simply not built yet. Each item says which it is, and how it shows up (or doesn't) after a copy. If one of these blocks you, tell us — support@nobo.dev — roadmap priority comes largely from these reports.

Content types

Inline comments — not copied (by design, for now)

Inline comments are anchored to exact text positions inside a page. There is no reliable way to re-anchor them onto a copied page — the anchor data doesn't survive the copy, and guessing produces comments attached to the wrong text. Copying them badly would be worse than not copying them, so we don't. How you'll know: copied pages simply have no inline comments; nothing in the report flags this, because it applies to every page uniformly.

Footer comments — copied only if you ask (optional, off by default)

Page (footer) comments can be carried over, but the option is off by default — turn on "Copy footer comments" in the copy options to include them. Not every copy wants the discussion thread duplicated, and keeping it opt-in keeps the permission footprint low (the comment scopes only matter when the toggle is used). When it is on:

  • Each source page's footer comments are recreated on the copy, and links inside comment bodies are rewritten the same way page links are, so an in-copy comment that referenced another copied page points at the copy.
  • Threaded replies are preserved as replies where the API accepts them; if a particular reply can't be threaded it is added as a top-level comment and the job notes that the threading was flattened.
  • Author attribution: Confluence's API authors every created comment as the app, so a copied comment cannot be posted as the original author. Instead each copied comment is prefixed with an attribution line — "Copied by Sure Copy — originally commented by @author on <date>" — mentioning the original author (so their name still shows) and making clear the app posted it.

Inline comments are still not copied (see above) — they can't be reliably re-anchored, so only footer comments are optional here. How you'll know: with the toggle off, copied pages have no comments; with it on, the completion report shows a "footer comments copied" count.

Whiteboards — not copied (Confluence API cannot read whiteboard content)

Whiteboards are detected and reported but not copied, and — after a direct investigation against the live Confluence Cloud API (ticket T-201) — copying them faithfully is not currently feasible, not merely unbuilt.

The blocker is on the read side. To copy a whiteboard we would have to read its canvas (shapes, sticky notes, connectors, text) from the source and recreate it in the destination. Confluence Cloud exposes no API that returns a whiteboard's content in any form:

  • The v2 GET /wiki/api/v2/whiteboards/{id} endpoint returns only metadata (id, title, parent, author, version) — there is no body field, and requesting ?body-format=… returns 404.
  • The v1 GET /wiki/rest/api/content/{id} body representations (storage, editor, view, export_view, atlas_doc_format, dynamic) all come back empty or as a placeholder empty-paragraph document for a whiteboard — even for a whiteboard that demonstrably has content. styled_view returns only page chrome/CSS, not the canvas.
  • There is no whiteboard export endpoint (…/whiteboards/{id}/export, /export/svg, etc. all 404).

The create side is comparatively open — POST /wiki/api/v2/whiteboards accepts a body in SVG, CSV, or HTML to import new author-supplied content (and lists WHITEBOARD_DOC_FORMAT, the native round-trip format, but it currently rejects it as "not currently supported"). This lets us create whiteboards, but only from content we author — it does not help us reproduce a source whiteboard, because the source content can't be read out in the first place.

Given that, the app deliberately does not create empty or partial whiteboard stubs. An empty whiteboard that looks copied would be worse than an honest "not copied" — it would imply the content came across when it didn't. So whiteboards are inventoried and reported (see below) and left uncopied until Atlassian ships a whiteboard content read/export API (or enables WHITEBOARD_DOC_FORMAT round-tripping), at which point this becomes buildable. How you'll know: the completion report itemises the whiteboard count as "not copied"; the destination space has no whiteboards.

Databases — not copied (Confluence API cannot read database content)

Confluence databases are detected and reported but not copied, and — after a direct investigation against the live Confluence Cloud API (ticket T-202) — copying them faithfully is not currently feasible, not merely unbuilt. The situation mirrors whiteboards: the blocker is the read side.

To copy a database we would have to read its schema (columns/field types) and its rows/entries from the source and recreate them in the destination. Confluence Cloud exposes no API that returns a database's content in any form. The entire public v2 database API surface is three operations — POST /wiki/api/v2/databases (create), GET /wiki/api/v2/databases/{id} (metadata), DELETE …/{id} (delete) — plus content-properties/operations/ancestors. None of them return the table's fields or rows:

  • GET /wiki/api/v2/databases/{id} returns metadata only (id, title, parent, author, version, space) — there is no body/fields/rows field, and ?body-format=… returns 404 (a deliberate "no body for this type", exactly like whiteboards).
  • The sub-resource routes you would expect — …/databases/{id}/entries, /rows, /columns, /fields, /schema — do not exist (they fall through to the Confluence SPA, i.e. no such REST route).
  • The v1 GET /wiki/rest/api/content/{id} body representations (storage, atlas_doc_format, view, export_view, editor, dynamic) all come back empty or as a placeholder empty-paragraph document for a database — the structured table content is never in any body. styled_view returns only page chrome/CSS, no data.
  • There is no database export endpoint in the API (…/databases/{id}/export, /export/csv, /rest/api/content/{id}/export/csv all 404). CSV/PDF export exists only in the Confluence UI, not the REST API.

The write side is comparatively open — the undocumented body field on POST /wiki/api/v2/databases exposes a CSV importer (it rejects an empty body with "No parseable database CSV mutations found — include at least one valid <fields>, <views>, or <data> section"), so a database's schema and rows can in principle be created from a CSV we author. But this does not help us reproduce a source database, because the source content can't be read out in the first place — there is nothing to feed the importer.

Given that, the app deliberately does not create empty database stubs. An empty database that looks copied would be worse than an honest "not copied" — it would imply the data came across when it didn't. So databases are inventoried and reported (see below) and left uncopied until Atlassian ships a database content read/export API. This is tracked upstream as CONFCLOUD-77328 ("Allow Manipulation of Confluence Databases using the REST API", Future Consideration); databases are also not preserved in Confluence backups, which is consistent with there being no content-level API for them. How you'll know: the completion report itemises the database count as "not copied"; the destination space has no databases.

Folders — copied (structure and nesting preserved)

Folders are a newer Confluence organisational content type — containers that group pages (and other folders) in the page tree. Sure Copy copies them. Unlike whiteboards and databases (whose content the API cannot read), a folder is essentially a title and a place in the tree, and both of those are fully readable and writable through the Confluence v2 folder API — so folders copy faithfully:

  • Every folder in the copied set is recreated in the destination (v2 POST /wiki/api/v2/folders).
  • Pages land under their correct destination folder — they are not flattened out of it.
  • Folder nesting is preserved — a subfolder inside a folder is recreated inside the destination copy of that folder, to any depth. Folders nested under a page (and pages nested under a folder) are handled too: the copy walks pages and folders as one combined tree, so a node's parent — page or folder — is always created before it.
  • The completion report counts the folders copied (e.g. "3 folders"), alongside pages, attachments and labels.

Folders are therefore no longer listed in the "not copied" inventory. How you'll know: the destination space shows the same folder structure as the source, with pages under the same folders; the completion report shows the folder count copied. (Manual sibling ordering within a folder is not preserved — see "Manual page ordering" below — but parent/child nesting is exact.)

Blog posts — copied (bodies, attachments, labels, links)

Blog posts are ordinary content many spaces use for announcements and updates. They share pages' storage format, attachments, labels and link model — differing only in that they are date-ordered rather than part of the page tree — so Sure Copy copies them on a whole-space copy, riding the same engine as pages:

  • Every blog post in the source space is recreated in the destination with its body, attachments and labels (v2 POST /wiki/api/v2/blogposts, enumerated via GET /wiki/api/v2/blogposts?space-id=…).
  • Links inside a blog body are rewritten exactly as for pages — a link to an in-set page (or blog) is re-pointed at the copy; a link outside the copied set is preserved and reported.
  • The copy is optional (a "Copy blog posts" toggle, on by default) and applies to whole-space (new-space) copies only — blog posts belong to a space, not to a page tree, so a "copy this page tree under a parent" job does not carry them.
  • The completion report counts the blog posts copied (e.g. "2 blog posts"), alongside pages, attachments, labels and folders.

Blog posts use the same read:/write:page scopes as pages — no additional scope is requested. When the toggle is on, blog posts are no longer listed in the "not copied" inventory; when it is off, they are inventoried and reported by count like any content the copy left behind. How you'll know: the destination space's blog shows the copied posts; the completion report shows the blog-post count copied (toggle on) or the count left behind (toggle off).

Not carried across: a blog post's original posting date and author are not preserved — each copy is created fresh (posted by the app, dated at copy time), the same way copied pages start a new version history. Blog-post comments follow the footer-comments option and behave as they do for pages. Links to a blog that are stored in the blog-specific ri:blog-post resource form (which encodes the post's title + posting-day) are not yet re-pointed at the copy; in-body page links and id/URL-based links are.

Drafts and version history

Only the current published version of each page is copied. Drafts are not copied, and the copy starts its version history from scratch — the source page's history does not come across. Pages are created in the destination as normal, current pages; an archived state on a source page is not preserved.

Space-level

Space permissions — not copied (by design)

The destination space is created with your site's default space permissions. Copying permissions would require admin-level scopes that the app deliberately does not request — a small permission footprint is part of the design (and part of why installation is easy to approve). What to do: set permissions on the new space after the copy, as you would for any new space. How you'll know: the new space behaves like any freshly created space.

Page restrictions — copied only if you ask (optional, off by default)

Per-page view/edit restrictions can be recreated on the copies, but the option is off by default — turn on "Recreate page restrictions" in the copy options to include them. It is opt-in for two reasons: it touches access control, and it is the one feature that needs a broader permission (see below). With the toggle off, copied pages get the destination's default access (today's behavior) and no restriction API call is made at all.

When it is on:

  • For each copied page, the source page's view and edit restrictions are read and the equivalent is applied to the copy — same users (by account) and same groups (by id/name). This is a same-site copy, so those principals are identical in the destination; no remapping is needed.
  • Any principal that can't be reproduced is reported on the job, never silently dropped. In practice, a valid source restriction copies cleanly; a principal is only unreproducible if it became invalid between the copy's read and write (for example a user deactivated in that window), in which case the job records exactly which view/edit principal was left off.
  • The Sure Copy app account is added to each restricted page it recreates. Confluence forbids the account performing a restriction change from locking itself out, and — unlike a person restricting their own page in the UI — group membership doesn't satisfy that rule for an app, so the app must list itself explicitly. This lets the app manage the copy; it grants no additional human any access (the real users/groups are exactly those from the source, plus the app).
  • Extra permission: recreating restrictions uses one additional scope, write:content.restriction:confluence (reading them needs no new scope). This is the single deliberate widening of the app's footprint, which is why it stays behind a toggle that is off unless you choose it. See the security page for the disclosure.
  • Permission check: when the toggle is on, the pre-copy check additionally verifies that you (the invoking user) actually hold the right to set restrictions on the source content (Confluence's restrict_content right). The app will not use its restriction scope to let a view-only user lock down pages they couldn't restrict themselves.

One inherent limit: Sure Copy can only copy a page it can read. A source page whose view restriction excludes the Sure Copy app entirely (for example, restricted to a handful of specific users with no group the app belongs to) is invisible to the app and therefore isn't copied at all — the same "can't copy what it can't read" behavior that applies to any page the app has no access to. Most restrictions (edit-only, or view restricted to a group the app can see) are unaffected. How you'll know: with the toggle off, copied pages carry no restrictions; with it on, the completion report shows a "page restrictions" count, and any principal it couldn't reproduce is listed in the job's notes.

Space description — copied (new-space mode)

When you copy a space into a brand-new space, the source space's description is carried over onto the new space. If the source has no description, the new space simply has none (no error). Copying a page tree into an existing space leaves that space's own description untouched — the existing space keeps its identity.

Space templates — copied (new-space mode)

When you copy a space into a brand-new space, the source space's user-created page templates come across — name, body, description and labels — so a team that clones a space to reuse its templates gets them. Each source template is recreated in the destination space (Confluence v1 POST /wiki/rest/api/template, listed with bodies via GET /wiki/rest/api/template/page?spaceKey=…&expand=body,labels). The completion report counts the templates copied (templatesCopied). Copying a page tree into an existing space does not touch that space's templates.

Two things about templates are not carried:

  • Promoted/customised blueprint templates (the system blueprints a space can promote, e.g. "HR Knowledge Base", "Meeting Notes") are not copied. These are plugin-owned scaffolds that reference a pluginKey/moduleKey rather than storing user content, so they can't be recreated as free-standing templates through the API. Only your own page templates copy.
  • Links inside a template body are copied verbatim (not rewritten). A template is a space-level scaffold, so a link embedded in one is rare; when present it keeps pointing at wherever it pointed in the source. (Page content links, by contrast, are rewritten — see Links below.)

Space theme / look-and-feel — not copied (no usable Cloud API)

The source space's theme and look-and-feel are not copied, and — spiked directly against the live Confluence Cloud API (ticket T-113) — this is not currently feasible, not merely unbuilt:

  • Confluence Cloud has deprecated selectable space themes. GET /wiki/rest/api/settings/theme returns an empty list — there are no installable/selectable themes to carry — and a normal space returns 404 "Theme for the Space cannot be found" on GET /wiki/rest/api/space/{key}/theme, i.e. it uses the product default. There is simply no theme value to copy.
  • A space's look-and-feel (GET /wiki/rest/api/settings/lookandfeel?spaceKey=…) reports selected: "global" — spaces inherit the site-wide look-and-feel; there is no space-scoped theme to reproduce.
  • The rare space that does set a custom colour scheme would require an admin-level configuration scope the app deliberately does not request (a small permission footprint is part of the design — the same reason space permissions aren't copied).

So the new space uses your site's default theme/look-and-feel — which, on modern Confluence Cloud, is what the source space uses too. How you'll know: the destination space looks like any new space on your site; the report never claims a theme was copied.

Space icon/logo — not copied (no public API)

The source space's icon/logo is not copied. This is not a design choice — Confluence Cloud provides no public REST API to set a space icon/logo (the space icon field is read-only; the gap is tracked upstream as CONFCLOUD-75358). The only way to set a space icon is the Confluence UI (Space Settings → Space Details). What to do: set the new space's icon by hand after the copy if you want it to match. How you'll know: the new space shows Confluence's default icon.

Space settings — not copied

Other space-level settings (sidebar configuration, space-level permissions overrides, etc.) are not copied. The new space gets the name and key you chose, the source's description and page templates (above), and otherwise your site's defaults. (Theme/look-and-feel has its own note above — it isn't copyable on Cloud.)

Links

/x/ short links — resolved when they point inside the copy

Short links (/wiki/x/AbCdE) are compact codes for a specific page. At copy time Sure Copy resolves each one to the page it targets (using that page's own tiny-link code, which the Confluence API returns alongside the page — no external lookup, no extra permissions):

  • Target is inside the copied set → the short link is rewritten to the canonical copy URL (/wiki/spaces/DST/pages/<newId>), so it points at the copy, not the original. No warning — it's a correct link.
  • Target is outside the copied set (another space, or a page not included in this copy) → left unchanged, with a warning recorded against the page so you know exactly where it is. It keeps working — it just points at the original page, which is the right behavior for an out-of-set link.

Resolutions are cached per copy, so a short link that appears many times costs nothing extra. See How copying works.

Links to pages outside the copy

Left untouched on purpose — they still point where they should. Not a limitation so much as a decision, but people ask.

Structural

Duplicate page titles in the source space

Confluence Cloud can allow two pages with the same title in one space. Sure Copy's resume-safety mechanism identifies already-copied pages by title, so if your source space has two pages with exactly the same title, the copy will contain one page for that title, and links to either original resolve to it. Title-based links to a duplicated title are rewritten with an "ambiguous: duplicate title" warning recorded on the job. If your space uses duplicate titles deliberately, rename one before copying.

Manual page ordering

Parent/child hierarchy is preserved exactly, but a manually dragged sibling order in the page tree is not — siblings appear in Confluence's default order in the copy.

Scope of the app

Copying between two Confluence sites — not supported

Sure Copy works within one Confluence Cloud site. Cross-site copying is a different security and permissions model entirely (your content would have to leave the site boundary, which this app never does — that guarantee is worth more than the feature). For cross-site synchronisation, other tools exist that specialise in it.

Copy modes — whole space, page tree into an existing space, or a single page (all supported)

Early versions could only copy a whole space into a brand-new space. That is no longer a limitation: Sure Copy now also copies a page tree into an existing space (placed under a parent page you choose), and a single page (with or without its descendants). A "Copy page tree to…" action on any page's ••• menu runs the same engine from the page itself. See the Options reference for how to pick a mode. (Cross-site copying, above, remains genuinely unsupported.)

How you'll know, in general

Every copy ends with a completion summary — pages, attachments, labels, folders, blog posts, and (when the option is on) footer comments — plus a per-page issues panel for anything that failed mid-copy, and recorded warnings for every link the rewriter deliberately left alone. The report also inventories the unsupported content types in the source space — whiteboards and databases — and tells you how many of each it found but did not copy. (Folders and blog posts are no longer in that list — they are copied; see "Folders" and "Blog posts" above. When the "Copy blog posts" option is turned off, blog posts return to that inventory as "not copied". Inline comments are still not itemised individually, because — see above — they are not copied at all, uniformly, on every page.)

Roadmap, in the order we currently intend it: detection-and-reporting of whiteboards/databases/folders (shipped) → footer comments option (shipped) → folder copy support (shipped) → blog posts (shipped). Whiteboard and database content copy are both blocked upstream — Confluence Cloud has no API to read a whiteboard's canvas or a database's schema/rows (see above); we will build each as soon as Atlassian ships a content read/export API. If your priority differs, say so: support@nobo.dev.
